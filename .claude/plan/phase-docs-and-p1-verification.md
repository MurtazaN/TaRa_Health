## Part A — Phase 1 Design: Verification Verdict

**Verdict: architecturally sound, NEEDS A REVISION PASS before implementation proceeds.**

The two-pipeline split (ingestion / answering), the `LLMClient` protocol seam, safety-separate-from-answering,
and the single-file SQLite+sqlite-vec storage are all correct and at the right altitude. The scaffold matches
§11 precisely. The issues below are **design-doc clarifications and contract tightening**, not a redesign.

### Consensus findings (flagged by ≥2 reviewers — strongest signal)

| ID | Severity | Finding | Where | Fix |
|----|----------|---------|-------|-----|
| V1 | CRITICAL | **Safety is the highest-stakes gap.** Emergency taxonomy is a non-exhaustive English keyword list (`RED_FLAG_PATTERNS`); "bias toward over-triggering" is asserted but never operationalized; no fail-closed contract if `screen()` errors; LLM confirmation marked "optional". | §3.3, `safety/triage.py`, `answering/answerer.py:27-29` | Specify a **required minimum emergency taxonomy** (add: seizure, LOC/unresponsive, sepsis/fever+confusion, one-sided numbness, throat/tongue swelling, poisoning/ingestion, pregnancy bleeding, meningitis signs, self-harm phrasing variants). Decision rule: **escalate if keyword OR LLM flags; LLM may never downgrade a keyword hit; on LLM error/timeout, fail toward escalation.** Make LLM confirmation non-optional (or explicitly scope Phase 1 to English). Guarantee a non-empty, locale-aware escalation message whenever `is_emergency`. |
| V2 | HIGH | **Citation provenance breaks at the `Citation` boundary.** Spans survive `ExtractedSpan → Chunk` but `Citation` carries only `chunk_id, filename, page` — `char_start/char_end` is dropped, so "open source page" links can't locate/highlight the cited text. | §4, `storage/models.py` Citation | Add `char_start/char_end` (+ optional snippet `text`) to `Citation`. | V3 | HIGH | **Embedding model/dim mismatch unvalidated + no re-index path.** `embed_dim=384` hard-coupled to `bge-small-en-v1.5`; `vec0` bakes the dim at creation; changing `TARA_EMBED_MODEL` silently corrupts search. | §3.2, `config.py`, `storage/vector.py`, `embeddings/embedder.py` | Validate `embed_dim == model.get_sentence_embedding_dimension()` at startup; persist the embed model+dim as metadata; define a re-embed/migration path when the model changes. |

| V8 | MEDIUM | **Framing post-check order/safety unspecified.** `apply_framing(raw)` runs on the raw string while citations are parsed separately → displayed text and citation markers can diverge; no fail-safe if framing raises. | §3.3/§6, `answering/answerer.py:44` | Specify: parse+map citations first, then **append-only / non-destructive** framing on one canonical string; framing failure still returns the grounded answer with a default disclaimer. |
| V9 | MEDIUM | **Audit log has no writer and is under-specified.** `queries` omits `retrieved_chunk_ids` (needed by §8 eval); nothing writes to `queries`; no `model_route` (local/hosted) so PHI egress isn't auditable. | §4/§7, `app.py`, `answerer.py` | Restore `retrieved_chunk_ids`; specify the writer; add `model_route` + egress timestamp to the record. |

### Single-reviewer findings worth folding into the revision

- **V10 (HIGH, fastapi):** PHI leaks via query params — `POST /ask` with bare `question: str` becomes `?question=...` in access logs/history. Use a Pydantic request body + response model. (`app.py`)
- **V11 (HIGH, fastapi):** Blocking sync calls (embed/LLM/OCR) inside async handlers stall the single-worker event loop for the full Ollama inference. Make handlers sync `def` (FastAPI threadpools them) or `run_in_executor`. (`app.py`)
- **V12 (HIGH, healthcare):** Honesty is prompt-only. Add a **numeric-grounding post-check** (monetary/lab figures in the answer must appear in a cited chunk) and a **low-retrieval-score → abstain** rule — a confidently wrong copay with a citation is worse than a refusal.
- **V13 (HIGH, healthcare):** **Safety recall is not a release gate** and §8 omits a **numeric/value-accuracy** eval dimension. Make safety recall gate at a target (e.g. 100% on the V1 taxonomy) *before* the safety layer is "done"; add value-accuracy eval. Note §10 currently builds the eval harness *last* (step 7) — move safety-recall measurement to land with the safety layer (step 4).
- **V14 (HIGH, architect):** Re-ingest idempotency / chunk-id stability undefined — re-upload duplicates everything and dangles historical citations. Add a content-hash dedup check, a deterministic `chunk_id` scheme (`{doc_id}:{page}:{char_start}`), and a replace-in-transaction policy.
- **V15 (MEDIUM, fastapi):** Config gaps — no `anthropic_api_key` field in `Settings` (forces a hard-coded env lookup, breaking the "config not code" guarantee); no `max_upload_bytes`; no `llm_timeout`. Add them + a `@model_validator` that fails loudly when `model_mode in (hosted, hybrid)` and the key is empty.
- **V16 (MEDIUM, both):** `datetime.utcnow()` is deprecated (3.12+) → `datetime.now(timezone.utc)`; pin an ISO-8601 UTC string convention. (`storage/models.py`)
- **V17 (MEDIUM, fastapi):** `ollama` response accessed as a dict (`resp["message"]["content"]`) but ollama ≥0.2 returns a typed object (`resp.message.content`) — will `TypeError` on first query. Verify against installed version. (`llm/local_ollama.py`)
- **V18 (MEDIUM, fastapi):** Two unacknowledged extraction tensions: (a) `pymupdf4llm` emits Markdown, not char-positioned text — accurate `char_start/char_end` needs raw PyMuPDF (`get_text("rawdict")`); (b) a chunk crossing a page boundary breaks the single `page: int` field. Resolve the `ExtractedSpan`/`Chunk` contract before implementing extraction/chunking.
- **V19 (LOW, both):** `init_db` creates the vector table on a *different connection* than the schema (fine for SQLite file-level — clarify the §3.2 rationale); `get_settings()` mkdir side-effects inside `@lru_cache` bite tests (move mkdir to `main()`); `blobs.save` trusts the user filename suffix (whitelist `.pdf/.png/.jpg/.jpeg/.tiff`); `check_same_thread=False` needed for threadpool handlers; context-budget cap needed so top-k chunks don't overflow qwen3:8b's window.

### Explicitly correct — do NOT change
Pre-check-before-answering hard short-circuit; safety as a separate testable module; `LLMClient` protocol seam (config-driven, lazy-imported); storage split in one file; provenance carried through `ExtractedSpan → Chunk`; §11 layout matches the tree.

## Part B — Phase 2–4 Documentation Plan

Source of truth for phasing is **PRD §11**. **One self-contained doc per phase** — each
`PHASE_N_TECHNICAL_DESIGN.md` is followable on its own when implementing that phase, with no
flipping between files. They mirror the structure of `PHASE_1_TECHNICAL_DESIGN.md` (Goals →
Architecture → Components → Data-model deltas → Flows → Safety → Privacy/Security → Testing →
Excludes → Build order → Repo-layout deltas) so the series reads coherently. The agent
foundations (orchestrator, tool protocol, confirmation gate, action audit, secrets) are
introduced **inside the Phase 2 doc** — Phase 2 is the first actions phase, so that is their
natural home; Phases 3 and 4 open with a short "Builds on" recap pointing at those sections
but otherwise stand alone for their own scope. **No separate shared foundations doc.**

### Docs to author (execution phase) — separate doc per phase

1. **`docs/PHASE_2_TECHNICAL_DESIGN.md`** — *First actions: Calendar + Reminders (self-contained).*
   This doc also introduces the agent foundations that later phases reuse:
   - **Orchestrator / planner loop** (`perceive → plan → confirm → act → observe`) — how a request decomposes into steps and selects tools; the planner *wraps* the Phase 1 answering+retrieval, doesn't replace them.
   - **Tool protocol** — a `Tool` abstraction analogous to the `LLMClient` protocol seam (name, JSON schema, `execute()`, dry-run/preview). Local-vs-remote (MCP) is a config decision, mirroring Phase 1's local/hosted switch.
   - **Confirmation gate** — hard gate before any external/financial effect; preview → explicit "yes" → execute; nothing consequential without confirmation (PRD §6.3, §9).
   - **Action audit + idempotency** — every action logged & user-viewable (PRD §8); idempotency keys so no double-book; extends the Phase 1 `queries` audit into an `actions` table.
   - **Secrets / OAuth token storage** — encrypted, on-device; reuses the `TARA_DB_KEY` encryption decision (ties back to V7).
   - **Safety interaction** — actions run *after* the Phase 1 safety pre-check; emergencies still short-circuit before any planning.
   - **Phase 2 scope:** Google Calendar **official MCP server** (OAuth2) as the first `Tool`; Apple Reminders / local notifications (device-bound, no universal cloud API). First end-to-end action flow through the confirmation gate (UC-3). Testing: gate enforcement, idempotency (no double event), OAuth failure handling.

2. **`docs/PHASE_3_TECHNICAL_DESIGN.md`** — *External actions: email, delivery, pharmacy (self-contained).*
   Opens with a "Builds on Phase 2" recap (orchestrator, tool protocol, confirmation gate, audit). Scope: Gmail draft+send (insurer/provider email, UC-2); Instacart IDP (**assisted handoff** — API returns a checkout link, doesn't place orders); CVS/Walgreens refill (gated, OAuth2, UC-6). Establishes the **assisted-handoff pattern** ("Tara prepares it, you confirm on their site") as the default for gated/financial APIs. Heightened confirmation for financial actions; secret-management hardening. Carries OPEN_QUESTIONS #3/#4 as decisions-to-make.

3. **`docs/PHASE_4_TECHNICAL_DESIGN.md`** — *Health portal + proactive (self-contained).*
   Opens with a "Builds on Phase 2/3" recap. Scope: Epic on FHIR / SMART on FHIR (OAuth2 patient auth) — **read (USCDI) first, scheduling write-back later** (OPEN_QUESTIONS #2; production needs Epic app review). Proactive prep & follow-ups — the first time Tara acts unprompted, so a new safety/consent surface (deterministic disclaimer, opt-in). Note read FHIR data may re-enter the Phase 1 ingestion/retrieval pipeline.

4. **Update `docs/PHASE_1_TECHNICAL_DESIGN.md`** — apply the Part A revision pass (V1–V19), bump to v0.3, and add a forward-reference to `PHASE_2_TECHNICAL_DESIGN.md` for the action seam.

5. **Update cross-references** — `docs/PRD.md` §11 links each phase to its new doc; `docs/OPEN_QUESTIONS.md` #2–#4 point at the phase doc that will resolve them; `CLAUDE.md` "What this is" notes the phase-doc series exists.

## Risks and Mitigation

| Risk | Mitigation |
|------|------------|
| Phase 2–4 docs over-specify before requirements firm up (YAGNI) | Keep them at the same "settled boundaries, open empirics" altitude as the Phase 1 doc; push unresolved choices to OPEN_QUESTIONS, not into the design. |
| Agent foundations duplicated across self-contained phase docs | Foundations are specified *once* in the Phase 2 doc; Phase 3/4 give a short "Builds on" recap + a link, not a re-spec — self-contained for their deltas without copy-paste drift. |
| Phase 1 revision changes stub contracts the scaffold relies on | Treat V1–V19 as doc + docstring changes only; no behavior implemented yet, so nothing to break. Flag any contract change explicitly in the stub docstring. |
| Doc-only multi-plan misread as license to implement code | This plan is documentation; production code stays untouched until a separate, explicit request. |
